# Network Intent Layer (NIL) — Specification 0.1.0-draft

**Status:** Draft (revision 5). **Release date:** 2026-06-11.
**Wire dialect:** `0.1` (the `nil` envelope const is unchanged). The reference Python package
ships **0.3.0**, which adds the `ROLLBACK` performative (§7.6) in place on this same `0.1`
dialect — no new schema namespace. The public kernel name is **SEQRD-PC**.
**This specification is implementation-independent.** It defines an open standard; the
reference implementation (wosool) demonstrates it but never defines it — conformance is
demonstrated against Annex B by any implementation equally.
The key words MUST, MUST NOT, REQUIRED, SHALL, SHOULD, MAY are to be interpreted as
described in RFC 2119 / RFC 8174.

---

## 1. Abstract
NIL is a constrained intermediate language for communication between untrusted autonomous
agents (Speakers) and systems of record for business operations (Systems). NIL separates the
probabilistic layer (understanding natural language, planning) from the deterministic layer
(execution under policy). Untrusted content can suggest; only well-formed NIL sentences that
pass policy can execute. NIL's design follows control-/data-flow separation (Debenedetti et
al., 2025 — CaMeL), structured delegation tokens (South et al., 2025), and the
plan-then-execute pattern (Beurer-Kellner et al., 2025).

## 2. Design principles (informative)
1. **Speaker-neutral.** One model, a swarm, or N cooperating agents are indistinguishable to
   the System. Authority attaches to Grants, never to architectures.
2. **Non-literal.** NIL has no keyword matching and no natural-language commands. Natural
   language exists at exactly two edges: agent input (any language, any dialect) and the
   human Preview (complete, natural, RTL-first where applicable).
3. **Safety is server-side.** No conformance property may depend on Speaker behavior, prompts,
   or model quality.
4. **Mistakes must be non-fatal.** The language cannot prevent a wrong sentence; it guarantees
   a wrong sentence cannot execute unreviewed beyond its tier.
5. **Deterministic-first.** Wherever a deterministic mechanism can answer (schema validation,
   identifier resolution against records, policy tables), it MUST run before — and bound — any
   model-based mechanism. Models fill gaps; they never overrule records.
6. **Fail-closed authority.** Every gate that controls spend, scope, or tier resolves errors
   toward refusal. An unavailable budget store means *not granted*, never *unlimited*.
7. **The System owns the claim.** What a Speaker is permitted to tell a human about an
   action's outcome is computed by the System from verified state (§11), never asserted by
   the Speaker.

## 3. Terminology
- **Speaker** — any agent emitting NIL sentences under a Grant.
- **System** — a conformant business backend (reference: wosool).
- **Owner** — the human principal of a Workspace.
- **Approval Surface** — a channel writable by the Owner and provably unwritable by Speakers.
- **Grant** — a scoped, budgeted, revocable authorization for a Speaker within a Workspace
  (§14, `schemas/grant.schema.json`).
- **Verb** — a domain action (e.g. `commerce.create_product`) defined in a Profile.
- **Profile** — a registered verb lexicon for a business domain (registry/profiles/).
- **Sentence** — one NIL message: envelope + performative + body.
- **Preview** — the human-readable rendering of a proposed action's complete material facts.
- **Refusal** — a structured negative `PROPOSAL` carrying a code from Annex A. Refusals are
  first-class outcomes, not errors: they are the language's way of saying *ask, don't guess*.
- **Candidate** — a System-minted `{id, name, source}` triple offered when a reference is
  ambiguous (§6.3).
- **Result** — the System-composed outcome envelope for an executed proposal (§11).
- **Decision Window** — the bounded period a parked proposal awaits the Owner (§7.4).
- **Suspension** — a System state in which Speaker execution is refused for a scope while a
  human is in control (§12).

## 4. Performatives (the closed verb set of the language itself)
| Performative | Plane | Direction | Semantics |
|---|---|---|---|
| `PROPOSE` | agent | Speaker→System | Request creation of an ActionProposal. Executes nothing. |
| `PROPOSAL` | agent | System→Speaker | The previewed, tiered, immutable proposal — or a Refusal. |
| `COMMIT` | agent | Speaker→System | Request execution of a proposal. Idempotency key REQUIRED. |
| `STATUS` | agent | both | Query/report the state of a proposal or sentence. |
| `QUERY` | agent | Speaker→System | Read-only question against granted read scopes. |
| `EVENT` | agent | System→Speaker | Asynchronous fact notification (§11.3). |
| `ROLLBACK` | agent | Speaker→System | Request reversal of a committed action. Executes nothing; the System answers with a `PROPOSAL` (the compensation preview), COMMITted like any action (§7.6). |
| `DECIDE` | **owner** | Approval Surface→System | Approve, reject, or approve-with-modification a parked proposal. **MUST NOT be acceptable from any Speaker credential.** |

The closed speaker-plane set is now **seven**: `PROPOSE, PROPOSAL, COMMIT, QUERY, STATUS,
EVENT, ROLLBACK` — mnemonic **SEQRD-PC** (S—STATUS · E—EVENT · Q—QUERY · R—ROLLBACK ·
D—DECIDE · P—PROPOSE · C—COMMIT; a memory aid, not a wire rename). `DECIDE` remains the
separate owner-plane performative. `ROLLBACK` is the backward counterpart of `PROPOSE`: it
closes the action lifecycle rather than extending it (6→7 is closure, not sprawl). The
performative set is closed; extension happens in Profiles (verbs), never here, and the set
itself grows only by ratified amendment. A System MUST reject unknown performatives.

## 5. The sentence envelope (normative)
Every sentence MUST validate against `schemas/envelope.schema.json`:
```json
{ "nil": "0.1", "id": "<uuid>", "performative": "PROPOSE",
  "grant": "<grant-id>", "workspace": "<ws-id>",
  "ts": "<RFC3339>", "trace": "<traceparent, optional>", "body": { } }
```
Rules:
- Unknown top-level members MUST be rejected (no silent tolerance — tolerance is how
  injection hides). `body` schema is selected by `performative`.
- All identifiers are opaque strings minted by the System. Identity facts (workspace, store,
  tenant) are bound to the Grant server-side; a Speaker-supplied identity value that
  contradicts the Grant MUST cause refusal of the whole sentence, never silent correction.
- `trace` carries a correlation identifier and SHOULD be a W3C Trace Context `traceparent`.
  Systems MUST propagate it through every downstream record (audit, events, telemetry) so one
  sentence is reconstructible end to end, and SHOULD additionally maintain a causation link
  (which sentence/event triggered this one) in their audit trail (§10).

## 6. PROPOSE / PROPOSAL semantics
A `PROPOSE` body: `{ "verb": "<profile.verb>", "args": { … } }`.
On receipt the System MUST, in order:

### 6.1 Authenticate and scope (G1)
Authenticate the Grant and check the verb against its scopes (default-deny). The Grant's
audience class (§14) restricts the reachable lexicon: a customer-audience Grant can never
reach owner-audience verbs regardless of scopes claimed.

### 6.2 Validate
Validate `args` against the verb's schema from the Profile. Type coercion is FORBIDDEN.
Exactly one transformation is permitted: **alias normalization** — a Profile MAY declare
argument aliases (e.g. `order_status` → `status`); when both canonical key and alias are
present the canonical key wins and the alias is discarded. Unknown argument members MUST be
rejected.

### 6.3 Resolve
Resolve all material values (amounts, parties, resources) against authoritative records.
Speaker-declared values for resolvable facts MUST be ignored or rejected. Resolution has
exactly three outcomes:

| Outcome | Meaning | System behavior |
|---|---|---|
| **RESOLVED** | exactly one authoritative match | proceed to §6.4 |
| **AMBIGUOUS** | ≥ 2 plausible matches | Refusal `AMBIGUOUS` carrying ≤ 8 Candidates `{id, name, source}` so the Speaker can ask the human to choose |
| **UNRESOLVED** | no match | Refusal `UNRESOLVED` — *ask, never guess* |

Normative resolution rules (all REQUIRED):
1. **Deterministic sources first.** Live records, then recently-observed entities, then
   session context. A model-based disambiguator MAY run only when deterministic tiers
   yield ≥ 2 candidates.
2. **Bounded disambiguation.** Any model-based disambiguator MUST be bounded in wall-clock
   time and input size, and on any failure (timeout, parse error, refusal) MUST degrade to
   the `AMBIGUOUS` refusal — never to a guess, never to a larger model.
3. **Fabrication defense.** A disambiguator may only *select among* System-provided
   Candidates. A selection outside the candidate set MUST be treated as disambiguator
   failure.
4. **Literal identifiers.** A Speaker-supplied value matching the System's identifier shape
   MAY pass resolution syntactically but MUST still be existence- and ownership-checked
   before execution; a well-formed identifier that does not exist in the Workspace is
   `UNRESOLVED`.
5. **Locale folding.** Matching SHOULD apply locale-aware normalization (case, diacritics,
   letter-form unification — e.g. Arabic alef/hamza variants, definite-article stripping)
   so that human references resolve the way humans write them.

### 6.4 Evaluate policy
Policy evaluation is an ordered pipeline; the first DENY refuses, later stages can only
maintain or raise the tier:
1. **Audience/lane** — is the verb reachable by this Grant's audience class?
2. **Capability** — is the verb enabled for this Workspace (e.g. subscription plan)?
3. **Workspace rules** — Owner-configured rules MAY deny or raise tier; they MUST NOT lower
   a tier below the Profile's floor.
4. **Amount thresholds** — resolved monetary exposure maps to tier per the Profile's
   escalation table (e.g. refund/discount above a configured limit ⇒ minimum HIGH).
5. **Tier assignment** — tier ∈ {LOW, MEDIUM, HIGH, CRITICAL}. Profiles MAY declare floors —
   minimum tiers a System MUST NOT allow to be lowered by any configuration.
6. **Origin escalation** — a System MAY apply stricter floors by sentence origin (e.g. a
   third-party Speaker via the MCP binding gets a higher floor than the first-party agent).
   Origin MUST NOT lower any tier.

Verbs marked `explicit_request: true` in a Profile additionally require that the proposal
originate from an explicit human request in the current interaction; the System MUST refuse
(`POLICY_DENIED`) speculative or self-initiated proposals of such verbs.

### 6.5 Render the Preview
The Preview MUST contain every material fact (actor, action, counterparty, amount+currency,
channel) in natural language. For verbs that mutate an existing entity, the Preview SHOULD
present the material change as *before → after*. Previews are keyed by BCP 47 language tag
and MUST include the Workspace's primary locale; for right-to-left locales (e.g. `ar`) RTL
rendering is REQUIRED. Monetary amounts carry ISO 4217 currency codes. All Preview values
MUST be rendered from resolved facts (§6.3), never from Speaker args. *An incomplete or
vague Preview is a conformance failure of the System, not a content style issue.*

### 6.6 Persist and answer
Persist the proposal immutably with an expiry (default 24h, equal to the default Decision
Window) and answer with `PROPOSAL` (`schemas/proposal.schema.json`) — either the previewed,
tiered proposal or a Refusal with a code from Annex A.

## 7. COMMIT semantics
`{ "proposal": "<id>", "idempotency_key": "<key>" }`. The System MUST:

### 7.1 Enforce idempotency (G4)
Same key ⇒ exactly-once effect, original outcome replayed with a `replayed: true` marker;
retention ≥ 48h. Keys are scope-qualified server-side (workspace + operation class) so a key
cannot collide across workspaces or operation kinds. Where the System writes through to an
upstream platform, it SHOULD propagate an idempotency token upstream; where the upstream
offers none, the System MUST guard with its own pre-write duplicate check and record the
residual risk in its conformance statement.

### 7.2 Enforce budgets (G5) BEFORE side effects
Three budget classes, all attached to the **Grant**:
- **action budgets** — count of executions per window;
- **monetary budgets** — resolved exposure per window, in the Workspace currency;
- **resource-class quotas** — named expensive capabilities (e.g. `reports.sales: 10/month`).

An absent budget means *not granted*, never unlimited. Budget checks MUST fail closed
(§2.6). A swarm sharing a Grant shares its budget and cannot multiply authority by
multiplying heads. Exhaustion is the Refusal `BUDGET_EXHAUSTED` (or `QUOTA_EXHAUSTED` for a
resource class) and MUST be emitted BEFORE any side effect.

### 7.3 Route by tier
- **LOW** ⇒ execute.
- **MEDIUM** ⇒ execute + notify Owner. The notification is approval-class traffic (§9):
  rate limits and quiet hours MUST NOT silently drop it — deliver, defer-with-record, or
  surface on the next Owner interaction.
- **HIGH** ⇒ park `pending_approval` + dispatch to the Approval Surface. The Decision
  Window defaults to 24h; a Profile MAY declare per-verb windows. On expiry the disposition
  MUST default to reject (`EVENT expired`); a Profile MAY declare `hold` for non-floor verbs;
  auto-approval on timeout MUST NOT exist for any verb with a floor of HIGH or above.
- **CRITICAL** ⇒ as HIGH, plus a cooling delay (default 300s) after approval before
  execution, during which the Owner can revoke.

### 7.4 DECIDE outcomes
A `DECIDE` (owner plane only, `schemas/decide.owner-plane.schema.json`) carries one of:
- **approve** — the parked proposal executes (after §7.3 cooling, if CRITICAL);
- **reject** — terminal; `EVENT rejected`;
- **approve-with-modification** — the decision carries replacement values for facts the
  Profile lists as `modifiable` for that verb (e.g. a discount percentage). The System MUST
  re-resolve, re-render the Preview, and re-evaluate tier with the modified values; if the
  tier of the modified action exceeds the original, the proposal re-parks rather than
  executing. Modification of non-modifiable facts MUST be rejected.

### 7.5 Report
Emit `EVENT` sentences for every state change (§11.3); record everything per §10.

### 7.6 ROLLBACK semantics (backward recovery)
`ROLLBACK` is the lifecycle-closing backward-recovery primitive — the backward counterpart of
`PROPOSE`. It does **not** execute a reversal; it *requests* one. A `ROLLBACK` body
(`schemas/rollback.schema.json`) carries:
- `compensation_token` (REQUIRED) — the token emitted on the original COMMIT's `EVENT` result
  (§11.1);
- `reason` (REQUIRED) — one of `saga_unwind | owner_cancel | downstream_failed | agent_repair`;
- `idempotency_key` (OPTIONAL).

On receipt the System resolves the token, determines the original verb's **reversibility
tier** (§14.2), and answers with a `PROPOSAL` — the *compensation preview* — which is then
`COMMIT`ted like any action. Because the reversal flows through PROPOSE→PROPOSAL→COMMIT, the
"no silent write" guarantee holds for free: no reversal executes unpreviewed. Outcomes by
tier:
- **REVERSIBLE** ⇒ the clean inverse is previewed (e.g. `create_product` ↔ `delete_product`);
- **COMPENSABLE** ⇒ an offsetting forward action is previewed (e.g. `record_payment` →
  `process_refund`);
- **IRREVERSIBLE** ⇒ Refusal `IRREVERSIBLE` — the System refuses to *pretend* it can undo;
- an unknown or expired token MUST NOT trigger a phantom reversal: unknown refuses, and an
  expired compensation window refuses `COMPENSATION_EXPIRED`.

Authority scales with tier: a low-risk REVERSIBLE reversal within the Grant lets an agent
self-heal; a COMPENSABLE or high-value reversal forces a human `DECIDE` (out-of-band, §7.4).
Auto-compensation of a failed multi-step program is permitted **only** for pre-blessed
REVERSIBLE steps named in the Grant's `auto_compensate` allowlist.

## 8. System conformance: the Six Guarantees (all REQUIRED)
- **G1** scoped default-deny Grants with audience classes;
- **G2** two-phase — no direct-write verb may exist in any Profile;
- **G3** out-of-band approval with credential separation — the DECIDE credential MUST NOT be
  issuable to, derivable by, or co-deployed with any Speaker credential;
- **G4** mandatory idempotency with outcome replay (§7.1);
- **G5** budgets-before-side-effects, fail-closed (§7.2);
- **G6** append-only audit binding every sentence to (grant, policy version, actor, preview
  shown) per §10.

A System that implements NIL syntax without all six guarantees MUST NOT claim NIL
conformance.

**Conformance classes.** This document defines **Core System** conformance (§1–§15). A
multi-tenant, operator-run service accepting NIL from more than one Workspace MUST
additionally satisfy the Hosted System profile — **NIL-H**, clauses H1–H8 in
[`versions/0.1.0-hosted-profile.md`](0.1.0-hosted-profile.md) (tenancy isolation, credential
lifecycle, rate limiting, signed push delivery, data protection, entitlement discovery,
tenant lifecycle, operational transparency) — to claim **NIL-H** conformance.

## 9. Approval Surface conformance
- MUST authenticate the Owner independently of any Speaker path (separate credential class:
  e.g. the Owner's verified channel identity or an authenticated dashboard session — never a
  service/API credential a Speaker could hold).
- MUST display the stored Preview verbatim.
- MUST transmit decisions with the owner-plane credential, and record decision actor and
  channel (e.g. `owner:whatsapp:+9665…`, `owner:dashboard`) in the audit trail.
- SHOULD support decision by reply on ubiquitous channels (e.g. WhatsApp) and SHOULD offer
  the approve / reject / modify affordances inline with the Preview.
- MUST verify channel deliverability before claiming dispatch: if the approval channel is
  known-down (e.g. a disconnected messaging instance), the System MUST NOT mark the proposal
  as awaiting decision silently — it MUST surface the delivery failure as an `EVENT` and via
  any secondary owner channel.
- Approval-class notifications MUST bypass quiet-hours and notification rate limits or be
  deferred *with a durable record and a visible pending marker* — never dropped.
- Confidence scores from any model MUST NOT be displayed as a basis for decision
  (miscalibration; see §15).

## 10. Audit requirements
Every sentence and transition MUST be recorded **append-only**: state changes are new
records, never in-place mutations of prior records. Each audit record MUST carry:

| Field | Content |
|---|---|
| `sentence_id` | the envelope `id` (or System-minted id for internal transitions) |
| `grant` | the Grant under which the sentence was accepted |
| `verb`, `tier` | resolved verb and final tier |
| `policy_version` | identifier of the policy/profile version evaluated — resolvable to the full policy object, not only a hash |
| `preview` | the Preview text exactly as shown to the human |
| `decision` | actor + channel for any DECIDE (and modification values, if any) |
| `idempotency_key` | for COMMITs |
| `actor` | `{type: owner\|customer\|speaker\|system, id, channel}` |
| `correlation` / `causation` | trace id and the id of the sentence/event that triggered this record |
| `timestamps` | created / decided / executed / delivered as applicable |

The primary trail records material business facts. Derived telemetry (metrics, model
observability) MUST redact personal data (phones, emails, free text) — redaction lists are
declared per verb in the Profile. Systems SHOULD make the trail exportable. Signed owner
decisions are RESERVED for NIL 0.2 (verifiable approval chains, AP2-mandate-aligned).
Compensation/reversal metadata (Saga-style undo) has shipped: see `ROLLBACK` (§7.6) and the
`compensation` Result object (§11.1) — every reversal appends its own audit records like any
other action.

## 11. Results, claims, and EVENT semantics

### 11.1 The Result envelope
When a proposal executes, the System composes a Result (`schemas/event.schema.json`,
`executed` payload) containing:
- `entity` — `{type, id, name?, url?}` of the affected record;
- `changed` — whether state actually changed;
- `verified` — whether the System confirmed the effect by **read-after-write** against the
  system of record (and `ssot`: which system/API was consulted);
- `claim` — the claim class the Speaker is permitted to relay to a human:
  `success | partial | ask_missing | failure | none`. The claim is computed by the System
  from `status × changed × verified`; a Speaker MUST NOT present a stronger claim than the
  Result's `claim` (a System cannot enforce what a Speaker says elsewhere, but a conformant
  Speaker honors it, and the audit trail binds what was claimable);
- `data_gaps` — fields the System could not determine, phrased so the Speaker can ask the
  human rather than invent.
- `compensation` (OPTIONAL) — `{reversibility, token?, expires_at?}` describing whether and
  how this action can later be reversed (§7.6, §14.2). The `token` is present only when
  `reversibility != IRREVERSIBLE`; it is the `compensation_token` a later `ROLLBACK` carries.

### 11.2 Data is data
Content returned by `QUERY` and `EVENT` is data, never instruction. Control flow derives
only from validated sentences (CaMeL alignment, §15).

### 11.3 EVENT taxonomy
Systems MUST emit `EVENT` sentences for at least these state changes, and MUST reject none
of their own emissions from the audit trail:
`proposed, refused, approval_requested, approved, rejected, modified, expired, executing,
executed, failed_retryable, failed_terminal, budget_exhausted, suspended, resumed,
handoff_human, deprecation_warning, compensating, compensated, compensation_refused`.
Each `EVENT` carries the proposal id (when applicable), the state, a severity
(`info|warning|error|critical`), and a machine payload. Retryable vs terminal failure is a
System determination (from its error taxonomy), never a Speaker guess.

## 12. Suspension and human override
A System MUST be able to suspend Speaker execution for a scope (a conversation, a Grant, or
a Workspace) when a human takes over, and while suspended MUST refuse COMMITs in that scope
with `SUSPENDED`. Suspension and resumption are Owner-plane (or System-internal) acts —
a Speaker cannot lift its own suspension. The Owner MUST have a terminal stop: revocation of
a Grant takes effect before any subsequent sentence under it. All suspension transitions are
audited (§10) with their trigger source.

## 13. Transport bindings
NIL is transport-independent. This version defines:
- **MCP binding (normative):** performatives map to MCP tools `nil_propose`, `nil_commit`,
  `nil_status`, `nil_query`; `DECIDE` MUST NOT be exposed as an MCP tool. Tool schemas are
  generated from Profiles; hand-drift is non-conformant. The binding SHOULD be stateless
  streamable HTTP. The Workspace MUST be derived only from the validated credential, never
  from a tool argument. Upstream platform credentials (e.g. a merchant's commerce-platform
  OAuth token) MUST NOT transit the binding in either direction. Read tools SHOULD carry
  `readOnlyHint: true`; propose tools return a proposal id and MUST NOT mutate.
- **HTTP binding (normative):** `POST /nil/v0.1/propose|commit|rollback`,
  `GET /nil/v0.1/status/{id}`, owner plane `POST /nil/v0.1/decide` under a distinct credential
  class. NIL exposes six endpoints: propose, commit, query, status/{id}, rollback, plus the
  events webhook (NIL-H H4). `POST /nil/v0.1/rollback` accepts a `ROLLBACK` body (§7.6) and,
  like propose, returns a `PROPOSAL` (the compensation preview) and MUST NOT mutate. Speaker
  authentication SHOULD follow OAuth 2.x bearer semantics (RFC 6750). Transport-level errors
  use RFC 9457 Problem Details; `RATE_LIMITED` maps to 429 + `Retry-After` (RFC 6585).
  Refusals are NOT transport errors — a well-formed sentence that refuses returns 200 with a
  Refusal body (the refusal is the protocol working, not failing).

## 14. Grants, Profiles & the Verb Registry

### 14.1 Grants
A Grant (`schemas/grant.schema.json`) binds: workspace; **audience class**
(`owner | customer | automation | third_party`); verb scopes (default-deny); the three
budget classes (§7.2); expiry; revocation state; and origin. Grants are minted and revoked
on the owner plane only.

### 14.2 Profile entries
Verbs live in Profiles (e.g. `commerce-v1`, `services-v1`, future `hospitality-v1`). A
Profile entry MUST define:
- **verb name + version** — verbs are individually semver-versioned
  (`commerce.create_product@1.1.0`); breaking arg changes bump major;
- **args schema** — with `enums`, `defaults`, and `aliases` (§6.2);
- **resolvable facts** — which args resolve server-side (§6.3) and which are
  identity-injected (never accepted from the Speaker at all);
- **base tier and floor** (if any), and the Profile's amount-escalation table;
- **flags** — `explicit_request` (§6.4), `destructive` (delete/irreversible class: floor
  HIGH minimum, MUST NOT be reachable by default scopes — a Grant must name destructive
  verbs explicitly), `idempotent_by_nature` (safe natural retries);
- **reversibility tier** (§7.6) — one of `REVERSIBLE` (a clean inverse exists, e.g.
  `create_product` ↔ `delete_product`), `COMPENSABLE` (no inverse but an offsetting forward
  action exists, e.g. `record_payment` → `process_refund`), or `IRREVERSIBLE` (no reversal;
  a `ROLLBACK` is refused honestly with `IRREVERSIBLE` at the preview). `IRREVERSIBLE` is the
  **default** for any unmarked verb (zero-touch backward compatibility) and is a strength, not
  a gap: the System refuses to pretend it can undo what it cannot;
- **modifiable facts** — which facts a DECIDE may modify (§7.4);
- **preview template keys** and **redaction list** (§10);
- **error contract** — which Annex A refusal codes and which failure classes the verb can
  produce;
- **deprecation** — `deprecated`, `replaced_by`; deprecated forms emit
  `EVENT deprecation_warning` for at least one MINOR release.

Profiles are registered via the process in GOVERNANCE.md. Speakers discover Profiles via
`QUERY { "verb": "nil.profiles" }`, which returns supported (spec, profile, verb-version)
tuples. Hosted Systems additionally serve `QUERY { "verb": "nil.capabilities" }` — the
Grant's effective entitlements, budgets-remaining, and limits (NIL-H, H6).

## 15. Security considerations
Threats and mandated mitigations:
- **hallucinated parameters** — server-side resolution (§6.3);
- **fabricated identifiers** — existence/ownership checks on literal ids (§6.3 rule 4) and
  the disambiguator fabrication defense (§6.3 rule 3);
- **prompt injection in retrieved data** — data returned by QUERY/EVENT is data, never
  instruction; control flow derives only from validated sentences (§11.2 — CaMeL alignment);
- **alias/argument smuggling** — closed schemas, alias normalization as the only
  transformation, canonical-key-wins (§6.2), unknown members rejected (§5);
- **identity confusion** — identity facts bound to the Grant; contradictions refuse (§5);
- **self-confirmation** — G3 credential separation; DECIDE unreachable from every Speaker
  binding (§4, §13);
- **origin spoofing** — Workspace and origin derived from the validated credential only
  (§13); origin can only raise tiers (§6.4);
- **retry storms** — G4 with outcome replay (§7.1) and bounded retry classes in the System's
  error taxonomy (§11.3);
- **runaway loops / swarm amplification** — G5 grant-attached budgets, fail-closed (§7.2);
- **overclaiming** — System-computed claim classes from read-after-write verification
  (§11.1);
- **notification fatigue as a bypass** — approval-class traffic exempt from silent
  suppression (§9);
- **repudiation** — G6 (§10).
Model confidence MUST NOT gate tier, autonomy, or be shown to deciders.

## Annex A — Refusal codes (normative)
A Refusal is a `PROPOSAL` with `outcome: "refusal"` and one code:

| Code | Stage | Meaning / required payload |
|---|---|---|
| `MALFORMED` | §5 | envelope or body fails schema; unknown members present |
| `UNKNOWN_PERFORMATIVE` | §4 | performative outside the closed set |
| `UNKNOWN_VERB` | §6.1 | verb not in any supported Profile version |
| `SCOPE_DENIED` | §6.1 | verb outside the Grant's scopes or audience class |
| `CAPABILITY_DENIED` | §6.4 | verb not enabled for this Workspace (e.g. plan) |
| `POLICY_DENIED` | §6.4 | workspace rule, threshold, or explicit-request rule denies |
| `INVALID_ARGS` | §6.2 | schema-valid envelope, invalid verb args (no coercion) |
| `UNRESOLVED` | §6.3 | no authoritative match; payload SHOULD name the unresolvable field |
| `AMBIGUOUS` | §6.3 | ≥2 matches; payload MUST carry Candidates (≤8) |
| `BUDGET_EXHAUSTED` | §7.2 | action or monetary budget exhausted (pre-side-effect) |
| `QUOTA_EXHAUSTED` | §7.2 | named resource-class quota exhausted; payload names class + window |
| `SUSPENDED` | §12 | scope under human override or Grant revoked |
| `EXPIRED` | §7 | COMMIT references an expired proposal |
| `IRREVERSIBLE` | §7.6 | ROLLBACK of a verb whose reversibility tier is `IRREVERSIBLE`; no reversal exists and none is pretended |
| `COMPENSATION_EXPIRED` | §7.6 | ROLLBACK whose `compensation_token` references an expired compensation window; no phantom reversal occurs |
| `RATE_LIMITED` | transport | transport-level throttling; retriable |
| `UPSTREAM_UNAVAILABLE` | §7 | system of record unreachable; retriable, no side effect occurred |

Refusal payloads are data for the Speaker to relay or act on (ask the human, choose a
candidate, stop) — never instructions.

## Annex B — Conformance checklist and implementation reports
The testable assertions of this specification (Core and NIL-H), the verification methods,
and the implementation-report process are maintained in
[`versions/0.1.0-conformance-checklist.md`](0.1.0-conformance-checklist.md). Conformance is
demonstrated by a published report, never declared. Per GOVERNANCE.md, an assertion the
reference implementation cannot pass blocks the non-draft release of its clause (implement
or demote), and 1.0 requires two independent interoperable implementations.
