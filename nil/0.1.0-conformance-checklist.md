# NIL 0.1.0-draft — Conformance Checklist and Implementation Reports (Annex B)

**Purpose.** NIL conformance is demonstrated, not declared. This annex enumerates the
testable assertions of the core specification and the NIL-H Hosted System profile, and
defines the implementation-report process. It is implementation-independent: assertions
reference clauses, never any particular codebase.

**Process (W3C/IETF-aligned).**
- An implementation claiming conformance publishes an **implementation report**: this
  checklist with a status per assertion (`pass | fail | n/a`), the verification method used,
  and logs of the `examples/` exchanges executed against it.
- GOVERNANCE.md applies: a normative clause must be demonstrated in the reference
  implementation before release, and any assertion the reference implementation cannot yet
  pass blocks a non-draft release of that clause (implement or demote).
- **Exit criterion for 1.0:** at least **two independent interoperable implementations**
  with passing reports (the W3C Recommendation pattern).

**Verification methods:** `N` = negative test (the forbidden thing must fail),
`F` = functional test, `I` = inspection of published artifacts/configuration.

## NIL-H Hosted System assertions

| ID | Clause | Assertion | Method |
|---|---|---|---|
| H-ISO-1 | H1.1–.3 | Every record, key, queue item, and candidate is scoped to one Workspace; shared-infrastructure identifiers carry the Workspace id and tenancy is re-checked at the execution boundary. | I/F |
| H-ISO-2 | H1.2 | A reference resolvable only in another Workspace refuses `UNRESOLVED`, indistinguishable (content and timing) from not-found. | N |
| H-ISO-3 | H1.4 | Same idempotency key in two Workspaces yields two independent effects. | F |
| H-ISO-4 | H1.5 | One tenant at its limits cannot starve another tenant's execution or approval dispatch. | F |
| H-CRD-1 | H2.1–.2 | Speaker credentials carry a distinct scannable prefix; per-credential revocation and two-key rotation work without downtime. | F |
| H-CRD-2 | H2.3 | A revoked Grant/credential refuses new sentences within 60s on every node. | F |
| H-CRD-3 | H2.4 | Credentials are encrypted at rest and absent from logs/traces/telemetry. | I |
| H-RTL-1 | H3 | Per-Grant and per-Workspace limits exist, are discoverable, refuse with `RATE_LIMITED` + retry-after, and fail closed on write paths; no sentence half-executes under load. | F/N |
| H-DLV-1 | H4 | Pushed EVENTs are signed (HMAC + timestamp), at-least-once with `delivery_id`, replay-skew rejected, retried into a drainable dead-letter store, per-proposal ordered, per-tenant isolated. | F |
| H-DAT-1 | H5.1 | Retention per record class is declared and published. | I |
| H-DAT-2 | H5.2 | The Owner can export the Workspace's audit trail and proposal history machine-readably. | F |
| H-DAT-3 | H5.3 | Verified erasure tombstones personal values in place; record structure, ordering, and ids survive; the tombstoning is audited and owner-plane only. | F |
| H-DAT-4 | H5.5 | TLS ≥ 1.2 on all bindings; hosting region(s) and trail jurisdiction published. | I |
| H-ENT-1 | H6.1 | Exactly one canonical entitlement resolver exists; all gates consult it. | I |
| H-ENT-2 | H6.2 | `nil.capabilities` returns the Grant's effective verbs, floors, budgets-remaining, limits, and retention values — and nothing about other tenants or global capacity. | F/N |
| H-ENT-3 | H6.4 | An unknown/unresolvable plan is treated as the most restrictive tier. | N |
| H-LIF-1 | H7 | No Speaker credential can exist without a revoking Owner; suspension refuses without deleting; offboarding runs revoke-all → export window → retention-driven deletion, each step audited. | F/I |
| H-OPS-1 | H8 | Deprecations carry dated sunsets ≥ one MINOR ahead with `deprecation_warning` events; health distinguishes System-down from `UPSTREAM_UNAVAILABLE`; isolation/credential incidents follow SECURITY.md disclosure. | I |
