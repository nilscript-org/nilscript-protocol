# 0.2.0 Conformance Appendix — Universal Connectability Evidence

> Informative. The proof that NIL is a **universal socket**, not a single-platform standard: any
> System that derives, splits, or abstracts the underlying concept can build a conforming adapter
> to the 0.2.0 lexicon **without faking**. Evidence: an 18-platform, 90-row calibration against
> **official vendor documentation only** (every row cites a fetched docs URL + verbatim quote).

## Method

5 representative verbs × 18 platforms, judged from each vendor's own API docs. Verdicts:
`clean` (direct), `adapter-compensates` (translation/multi-step), `gap` (cannot without faking).
Gap types: **(A)** platform deficiency, **(B)** standard-leak (the verb encoded one platform's
mechanism), **(C)** capability the standard had not defined.

## Platforms (category-diverse)

| Category | Platforms |
|---|---|
| ecommerce / OSS commerce | WooCommerce, Shopify, Saleor, Medusa, Vendure, Sylius |
| ERP | ERPNext, Odoo, Tryton |
| billing / subscription / MoR | Stripe Billing, Square, Chargebee, Recurly, Lemon Squeezy |
| scheduling | Cal.com, Calendly, Bookeo, SimplyBook.me |

## What the calibration proved (and what 0.2.0 does about it)

| Finding | Evidence | 0.2.0 response |
|---|---|---|
| **Order status is derived, not set** | 17/18 platforms derive status or gate it behind a state machine; only WooCommerce sets it freely | §A — `record_fulfillment`/`record_payment` (type-B resolved) |
| **The refund unit is platform-dependent** | Stripe/Square refund a payment; Chargebee/Recurly/ERPNext/Tryton/Odoo an invoice | §B — `refund_target {order\|invoice\|payment}` (type-B resolved) |
| **Product/variant/price/inventory are split** | ~10 platforms decompose; flat is clean only on WooCommerce/ERPNext/Odoo | §C — `variants[]` oneOf (type-B/C resolved) |
| **Scheduling is an undefined capability class** | Cal.com/Calendly/Bookeo/SimplyBook have first-class booking APIs with no NIL verb | GAP-004 — **deferred** declared-design (no reference implementer) |

## Cumulative verdicts (90 rows)

16 clean · 34 adapter-compensates · 40 gap. The gaps cluster on (1) `update_order_status` (now
replaced) and (2) the four scheduling platforms (a deferred domain). After 0.2.0, the remaining
commerce/services verbs are satisfiable by every non-scheduling platform via a conforming adapter:

- `create_coupon` — most portable (6 clean; its only gaps are platforms with no commerce at all).
- `create_invoice` — clean on every ERP (3/3); order-derived or absent elsewhere (a platform trait).
- `create_product` / `process_refund` — clean or adapter-compensable everywhere post-fix.

## Conclusion

A System does **not** have to model orders, a flat product, or an order-keyed refund to conform —
it adapts its *own* model (derived status, split catalog, invoice/payment refunds) to the socket.
That is the definition of a universal socket: the device adapts to the port, not the reverse. The
full row-level matrix lives in the reference implementation's calibration record
(`wosool-cloud/docs/calibration/`).
