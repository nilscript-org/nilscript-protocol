# NIL-H — Hosted System Profile, 0.1.0-draft

**Status:** Draft (normative companion to [the core specification](0.1.0.md)).
**Release date:** 2026-06-11. The key words MUST, MUST NOT, REQUIRED, SHALL, SHOULD, MAY are
to be interpreted as described in RFC 2119 / RFC 8174.

## Conformance classes

| Class | Requirements | Who claims it |
|---|---|---|
| **Core System** | core spec §1–§15 (the Six Guarantees) | any single-tenant or embedded deployment |
| **Hosted System (NIL-H)** | Core **plus** H1–H8 below | any multi-tenant, operator-run service accepting NIL from more than one Workspace |

A multi-tenant deployment that satisfies the Six Guarantees but not H1–H8 MUST NOT claim
NIL-H conformance. Every H-clause exists because multi-tenancy converts a single-tenant
inconvenience into a cross-customer breach, a noisy-neighbor outage, or a compliance failure.

---

## H1 — Tenancy and isolation
1. The **Workspace is the isolation unit.** Every sentence, proposal, Grant, idempotency
   key, audit record, cached entity, queue item, and resolution candidate MUST be scoped to
   exactly one Workspace.
2. **Resolution cannot cross tenants.** Candidates (§6.3) MUST be drawn only from the
   sentence's Workspace. A literal identifier that exists in *another* Workspace is
   `UNRESOLVED` — its existence MUST NOT be observable (no "exists but not yours"
   distinction in any refusal).
3. **Shared infrastructure MUST scope identifiers** — durable-execution workflow ids,
   cache keys, counters, and pointers carry the Workspace id; tenant attribution is
   re-checked server-side at the execution boundary, not only at ingress.
4. **Idempotency is tenant-scoped** (§7.1): a key can never collide — or replay — across
   Workspaces.
5. **Fairness.** A Hosted System MUST bound the share of execution capacity one Workspace
   can consume, so one tenant's swarm cannot starve another's Decision Window dispatches.
6. **Compensation tokens are tenant-scoped.** A `compensation_token` (core §7.6) MUST resolve
   only within its own Workspace; a `ROLLBACK` (`POST /nil/v0.1/rollback`) carrying a token
   minted in another Workspace is `UNRESOLVED`, indistinguishable from not-found (as H1.2).
   Authority scaling holds per tenant: REVERSIBLE reversals within grant self-heal, while
   COMPENSABLE/high-value reversals force that Workspace's own out-of-band `DECIDE`, and the
   `auto_compensate` allowlist is per-Grant — one tenant's allowlist never widens another's.

## H2 — Credential and Grant lifecycle
1. Speaker credentials MUST carry a recognizable, vendor-distinct prefix (e.g. `nil_sk_…`)
   so secret scanners can identify leaks, and MUST be revocable individually.
2. **Rotation without downtime:** a Hosted System MUST allow two active credentials per
   Grant during rotation windows.
3. **Revocation propagates fast:** a revoked Grant or credential MUST refuse new sentences
   within 60 seconds across all nodes; cached authorization state MUST have a TTL ≤ that
   bound.
4. Credentials and upstream platform tokens MUST be encrypted at rest (vaulted), MUST NOT
   appear in logs, traces, or telemetry (assert presence as booleans, never values), and
   MUST NOT transit any NIL binding (§13).
5. The DECIDE credential remains channel-bound owner identity or an authenticated owner
   session (§9); a Hosted System MUST NOT mint API-style DECIDE credentials at all.

## H3 — Rate limiting and backpressure
1. Limits exist at three layers — per-Grant, per-Workspace, and global — and the per-Grant
   and per-Workspace limits MUST be discoverable via `nil.capabilities` (H6).
2. Throttling answers with the Refusal `RATE_LIMITED` and a retry-after hint; in the HTTP
   binding this maps to `429` + `Retry-After`.
3. Rate-limit checks on write paths MUST fail closed (core §2.6). Backpressure MUST engage
   *before* side effects: a sentence is either refused whole or processed whole — never
   half-executed under load.
4. Limit changes are configuration, not policy: they MUST NOT lower any tier, floor, or
   budget (limits bound throughput, Grants bound authority).

## H4 — EVENT push delivery
Where EVENTs are pushed to Speaker-registered endpoints (webhooks) rather than polled
(delivery mechanics SHOULD follow the *Standard Webhooks* conventions):
1. **At-least-once with consumer dedupe:** every delivery carries a `delivery_id` and the
   `event` sentence id; consumers dedupe on sentence id, and redeliveries are byte-identical.
2. **Signed:** deliveries MUST be signed (HMAC with a per-endpoint secret) over body +
   timestamp; consumers MUST be able to reject replays older than a declared skew.
3. **Retried, then parked:** failed deliveries retry with exponential backoff and bounded
   attempts, then land in a dead-letter store that is *drainable* — redelivery after repair
   MUST be possible and MUST remain idempotent end to end.
4. **Ordered where it matters:** per-proposal state changes MUST be reconstructible in
   order (monotonic sequence per proposal); global ordering is NOT promised.
5. Delivery failure to one tenant's endpoint MUST NOT delay another tenant's deliveries
   (per-tenant delivery queues or equivalent isolation).

## H5 — Data protection (PDPL/GDPR-aligned)
1. **Declared retention** per record class. Recommended defaults: audit trail ≥ 1 year;
   idempotency outcomes ≥ 30 days; raw event/ingest logs ≤ 180 days; observability
   payloads ≤ 90 days. A Hosted System MUST publish its actual values.
2. **Export:** the Workspace's audit trail and proposal history MUST be exportable in a
   machine-readable form on Owner request (core §10's SHOULD becomes MUST under NIL-H).
3. **Erasure vs append-only:** on a verified erasure request, personal data fields inside
   audit records are **pseudonymized in place by tombstone** (the record, its ordering, ids,
   amounts, and integrity remain; the personal values are irreversibly replaced and the
   tombstoning is itself an audited act). Append-only is a structural property of records,
   not a license to retain personal data forever.
4. **Telemetry redaction** (core §10) is REQUIRED, driven by each verb's Profile redaction
   list — phones, emails, and free text never reach metrics or model-observability stores.
5. **Encryption:** TLS ≥ 1.2 on every binding; credentials and platform tokens encrypted at
   rest (H2.4). A Hosted System MUST publish its hosting region(s) and the jurisdiction its
   trail is subject to (residency statement).

## H6 — Entitlements and discovery
1. Capability gating (core §6.4 stage 2) is keyed to the Workspace's subscription plan and
   MUST resolve through **one canonical resolver** — duplicated ad-hoc entitlement lookups
   drift, and drift here is a billing-security failure; they are non-conformant.
2. Reserved discovery verbs (read plane, data-not-instruction):
   - `nil.profiles` — supported (spec, profile, verb-version) tuples (core §14.2);
   - `nil.capabilities` — this Grant's effective entitlements: enabled verbs, tier floors in
     effect, budgets and quotas with **remaining** amounts and windows, rate limits (H3),
     declared retention values (H5), each verb's **reversibility tier** (core §7.6/§14.2),
     and the Grant's `auto_compensate` allowlist (the pre-blessed REVERSIBLE steps a failed
     multi-step program may self-compensate).
3. Plan changes take effect within one cache window (≤ 60s, H2.3). A suspended Workspace
   refuses all writes with `SUSPENDED` and SHOULD refuse reads beyond `nil.capabilities`
   (so a Speaker can still learn *why*).
4. An unknown or unresolvable plan MUST be treated as the most restrictive tier, never as a
   default-open.

## H7 — Tenant lifecycle
1. **Provisioning** mints the Workspace's initial owner-plane identity before any Grant can
   exist; there is no window in which a Speaker credential exists without an Owner who can
   revoke it.
2. **Suspension** (billing, abuse, Owner request) refuses new sentences (`SUSPENDED`),
   parks nothing new, and lets in-flight Decision Windows expire normally. Suspension MUST
   NOT delete data.
3. **Offboarding** is ordered: revoke all Grants (cascade, H2.3) → export window for the
   Owner (H5.2) → deletion per declared retention (H5.1). Each step is audited; the
   Grant-revocation cascade MUST complete before any data deletion begins.

## H8 — Operational transparency
1. **Deprecation with dates:** deprecated verbs/spec versions emit
   `EVENT deprecation_warning` (core §14.2) and a published sunset date at least one MINOR
   release ahead (VERSIONING.md).
2. **Health:** each binding exposes a liveness/readiness signal suitable for Speaker-side
   failover; `UPSTREAM_UNAVAILABLE` refusals MUST be distinguishable from System-down.
3. **SLOs:** a Hosted System SHOULD publish latency objectives for PROPOSE and COMMIT and
   availability for the Approval Surface dispatch path — the Decision Window only protects
   the Owner if the approval request actually arrives.
4. **Incident disclosure:** security incidents affecting tenant isolation (H1) or
   credential confidentiality (H2) follow the coordinated-disclosure commitments in
   SECURITY.md toward affected Owners.

## Security considerations (delta to core §15)
- **Cross-tenant inference** — H1.2 makes other-tenant existence unobservable; refusal
  codes and timing SHOULD NOT distinguish "not yours" from "not anywhere".
- **Leaked credentials** — H2.1 prefixes enable scanning; H2.3 bounds blast-radius time.
- **Replayed webhooks** — H4.2 timestamps + skew rejection.
- **Quota probing** — `nil.capabilities` reveals a Grant's own limits only; it MUST NOT
  reveal global or other-tenant capacity.
- **Erasure abuse** — H5.3 tombstoning is owner-plane, verified, and audited; it cannot be
  used by a Speaker to launder the trail.
